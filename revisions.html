SQL 01 Langage de Définition des Données (LDD)

Le LDD (ou DDL en anglais — Data Definition Language) permet de créer, modifier et supprimer des structures de base de données : tables, colonnes, contraintes, index…

CREATE DATABASE — Créer une base

CREATE DATABASE ma_base
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;

-- Sélectionner la base à utiliser
USE ma_base;

CREATE TABLE — Créer une table complète

CREATE TABLE clients (
  id_client     INT           NOT NULL AUTO_INCREMENT,
  nom           VARCHAR(100)  NOT NULL,
  prenom        VARCHAR(100)  NOT NULL,
  email         VARCHAR(200)  NOT NULL UNIQUE,
  date_naissance DATE,
  age           INT           CHECK (age >= 0 AND age <= 150),
  ville         VARCHAR(100)  DEFAULT 'Lyon',
  actif         BOOLEAN       DEFAULT TRUE,
  PRIMARY KEY (id_client)
);

CREATE TABLE commandes (
  id_commande   INT           NOT NULL AUTO_INCREMENT,
  id_client     INT           NOT NULL,
  date_commande DATE          NOT NULL,
  montant       DECIMAL(10,2) NOT NULL CHECK (montant > 0),
  PRIMARY KEY (id_commande),
  FOREIGN KEY (id_client) REFERENCES clients(id_client)
    ON DELETE CASCADE
    ON UPDATE CASCADE
);
💡
Les contraintes d'intégrité :
• PRIMARY KEY : identifiant unique de chaque ligne, non null
• FOREIGN KEY : référence vers une clé primaire d'une autre table
• NOT NULL : la valeur est obligatoire
• UNIQUE : pas de doublon dans la colonne
• CHECK (condition) : valeur doit respecter une condition
• DEFAULT valeur : valeur par défaut si rien n'est précisé
⚠️
ON DELETE / ON UPDATE :
• CASCADE : supprime/modifie automatiquement les enregistrements liés
• SET NULL : met la clé étrangère à NULL
• RESTRICT (défaut) : bloque l'opération si des enregistrements liés existent
• NO ACTION : similaire à RESTRICT

ALTER TABLE — Modifier une table existante

-- Ajouter une colonne
ALTER TABLE clients
  ADD COLUMN telephone VARCHAR(20);

-- Modifier le type d'une colonne
ALTER TABLE clients
  MODIFY COLUMN nom VARCHAR(200) NOT NULL;

-- Renommer une colonne
ALTER TABLE clients
  RENAME COLUMN telephone TO tel;

-- Supprimer une colonne
ALTER TABLE clients
  DROP COLUMN tel;

-- Ajouter une contrainte
ALTER TABLE clients
  ADD CONSTRAINT uk_email UNIQUE (email);

-- Supprimer une contrainte (clé étrangère)
ALTER TABLE commandes
  DROP FOREIGN KEY fk_client;

-- Renommer la table
RENAME TABLE clients TO utilisateurs;

DROP & TRUNCATE — Supprimer

-- Supprimer une table (structure + données)
DROP TABLE IF EXISTS commandes;
DROP TABLE IF EXISTS clients;

-- Supprimer toute la base de données
DROP DATABASE IF EXISTS ma_base;

-- Vider une table (garde la structure, réinitialise l'auto-increment)
TRUNCATE TABLE commandes;

INDEX — Accélérer les recherches

-- Créer un index simple
CREATE INDEX idx_nom ON clients(nom);

-- Créer un index unique
CREATE UNIQUE INDEX idx_email ON clients(email);

-- Supprimer un index
DROP INDEX idx_nom ON clients;
SQL 02 LMD sans SELECT — INSERT, UPDATE, DELETE

Le LMD (Langage de Manipulation des Données / DML) permet d'insérer, modifier et supprimer des données.

INSERT INTO — Insérer des données

-- Insérer une seule ligne (avec noms de colonnes)
INSERT INTO clients (nom, prenom, email, ville)
VALUES ('Dupont', 'Jean', 'jean.dupont@mail.fr', 'Paris');

-- Insérer plusieurs lignes en une seule requête
INSERT INTO clients (nom, prenom, email)
VALUES
  ('Martin', 'Alice', 'alice@mail.fr'),
  ('Bernard', 'Bob', 'bob@mail.fr'),
  ('Petit', 'Clara', 'clara@mail.fr');

-- Insérer depuis une autre table (INSERT ... SELECT)
INSERT INTO archives_clients (nom, email)
SELECT nom, email
FROM clients
WHERE actif = FALSE;
💡
Si tu n'indiques pas les noms de colonnes, tu dois fournir des valeurs pour toutes les colonnes dans l'ordre exact de la table. Toujours préciser les colonnes = bonne pratique !

UPDATE — Modifier des données

-- Mettre à jour UN client précis
UPDATE clients
SET ville = 'Lyon', actif = TRUE
WHERE id_client = 5;

-- Augmenter de 10% tous les montants de commandes du mois dernier
UPDATE commandes
SET montant = montant * 1.10
WHERE MONTH(date_commande) = MONTH(NOW()) - 1;

-- UPDATE avec JOIN (modifier via une autre table)
UPDATE commandes c
JOIN clients cl ON c.id_client = cl.id_client
SET c.montant = c.montant * 0.9
WHERE cl.ville = 'Paris';
🚨
DANGER ! Un UPDATE sans WHERE met à jour TOUTES les lignes de la table. Toujours vérifier ta clause WHERE avant d'exécuter !

DELETE — Supprimer des données

-- Supprimer un client précis
DELETE FROM clients
WHERE id_client = 42;

-- Supprimer tous les clients inactifs
DELETE FROM clients
WHERE actif = FALSE;

-- DELETE avec sous-requête
DELETE FROM commandes
WHERE id_client IN (
  SELECT id_client FROM clients
  WHERE ville = 'Bordeaux'
);

-- Vider toute la table (utiliser TRUNCATE si pas besoin de rollback)
DELETE FROM clients; -- ⚠️ supprime TOUT

Transactions — Sécuriser les opérations

START TRANSACTION;

UPDATE comptes SET solde = solde - 500 WHERE id = 1;
UPDATE comptes SET solde = solde + 500 WHERE id = 2;

-- Si tout va bien :
COMMIT;

-- Si une erreur survient :
ROLLBACK;
💡
Les transactions garantissent les propriétés ACID : Atomicité, Cohérence, Isolation, Durabilité. Soit tout passe, soit rien ne passe.
SQL 03 LMD avec SELECT + Fonctions MySQL

Structure de base d'un SELECT

SELECT   [colonnes ou expressions]      -- 1. Quoi afficher
FROM     [table(s)]                      -- 2. D'où
JOIN     [autre table] ON [condition]   -- 3. Jointures
WHERE    [condition de filtrage]         -- 4. Filtrer les lignes (avant GROUP BY)
GROUP BY [colonne(s)]                   -- 5. Regrouper
HAVING   [condition sur agrégat]        -- 6. Filtrer les groupes (après GROUP BY)
ORDER BY [colonne] [ASC|DESC]           -- 7. Trier
LIMIT    [n] OFFSET [m];               -- 8. Pagination
⚠️
L'ordre d'écriture est obligatoire. WHERE ≠ HAVING : WHERE filtre les lignes avant le regroupement, HAVING filtre les groupes après l'agrégation.

SELECT — Les bases

-- Tout afficher
SELECT * FROM clients;

-- Colonnes spécifiques avec alias
SELECT
  nom    AS 'Nom de famille',
  prenom AS 'Prénom',
  ville
FROM clients
WHERE actif = TRUE
ORDER BY nom ASC;

-- DISTINCT : éliminer les doublons
SELECT DISTINCT ville FROM clients;

-- LIMIT + OFFSET : pagination (page 2, 10 résultats par page)
SELECT * FROM clients
ORDER BY nom
LIMIT 10 OFFSET 10;

Opérateurs de filtrage (WHERE)

-- Comparaison
WHERE age >= 18 AND age <= 65

-- BETWEEN (inclusif)
WHERE montant BETWEEN 100 AND 500

-- IN (liste de valeurs)
WHERE ville IN ('Paris', 'Lyon', 'Bordeaux')

-- LIKE (pattern) : % = n'importe quels caractères, _ = un seul
WHERE email LIKE '%@gmail.com'
WHERE nom LIKE 'D_pont'          -- Du, Di, Da...

-- IS NULL / IS NOT NULL
WHERE date_naissance IS NULL
WHERE telephone IS NOT NULL

-- NOT
WHERE ville NOT IN ('Paris', 'Lyon')

Les Jointures — Comprendre et maîtriser

-- INNER JOIN : seulement les lignes qui correspondent des 2 côtés
SELECT c.nom, c.prenom, cmd.montant
FROM clients c
INNER JOIN commandes cmd ON c.id_client = cmd.id_client;

-- LEFT JOIN : tous les clients, même sans commande
SELECT c.nom, cmd.montant
FROM clients c
LEFT JOIN commandes cmd ON c.id_client = cmd.id_client;

-- RIGHT JOIN : toutes les commandes, même sans client trouvé
SELECT c.nom, cmd.montant
FROM clients c
RIGHT JOIN commandes cmd ON c.id_client = cmd.id_client;

-- Jointure sur 3 tables
SELECT c.nom, p.libelle, l.quantite
FROM clients c
JOIN commandes cmd ON c.id_client = cmd.id_client
JOIN lignes_commande l ON cmd.id_commande = l.id_commande
JOIN produits p ON l.id_produit = p.id_produit;
Type de jointureRésultatCas d'usage
INNER JOINIntersection (correspondances des 2 côtés)Clients ayant commandé
LEFT JOINToutes les lignes de gauche + correspondances droite (NULL si absent)Clients avec ou sans commande
RIGHT JOINToutes les lignes de droite + correspondances gaucheCommandes avec ou sans client
CROSS JOINProduit cartésien (toutes les combinaisons)Générer des paires

Fonctions d'agrégation + GROUP BY / HAVING

-- Fonctions d'agrégation de base
SELECT
  COUNT(*)             AS total_clients,
  COUNT(email)         AS clients_avec_email,  -- ignore les NULL
  SUM(montant)          AS chiffre_affaires,
  AVG(montant)          AS panier_moyen,
  MIN(montant)          AS plus_petite_commande,
  MAX(montant)          AS plus_grande_commande
FROM commandes;

-- GROUP BY : chiffre d'affaires par ville
SELECT
  c.ville,
  COUNT(cmd.id_commande) AS nb_commandes,
  SUM(cmd.montant)       AS total
FROM clients c
JOIN commandes cmd ON c.id_client = cmd.id_client
GROUP BY c.ville
HAVING SUM(cmd.montant) > 10000    -- seulement les villes > 10 000€
ORDER BY total DESC;

Sous-requêtes (Subqueries)

-- Sous-requête dans WHERE
SELECT * FROM clients
WHERE id_client IN (
  SELECT id_client FROM commandes
  WHERE montant > 1000
);

-- Sous-requête avec EXISTS
SELECT * FROM clients c
WHERE EXISTS (
  SELECT 1 FROM commandes cmd
  WHERE cmd.id_client = c.id_client
);

-- Sous-requête scalaire dans SELECT
SELECT
  nom,
  (SELECT COUNT(*) FROM commandes cmd
   WHERE cmd.id_client = c.id_client) AS nb_commandes
FROM clients c;

UNION — Combiner des résultats

-- UNION : combine et supprime les doublons
SELECT nom, email FROM clients
UNION
SELECT nom, email FROM fournisseurs;

-- UNION ALL : combine ET garde les doublons (plus rapide)
SELECT nom FROM clients
UNION ALL
SELECT nom FROM fournisseurs;
💡
Les requêtes combinées doivent avoir le même nombre de colonnes avec des types compatibles. Le nom des colonnes du résultat est celui de la première requête.
SQL 04 LCD — Triggers (Déclencheurs)

Un trigger est un bloc de code SQL qui s'exécute automatiquement en réponse à un événement (INSERT, UPDATE, DELETE) sur une table.

Syntaxe d'un Trigger

CREATE TRIGGER nom_trigger
  [BEFORE | AFTER] [INSERT | UPDATE | DELETE]
  ON nom_table
  FOR EACH ROW
BEGIN
  -- Corps du trigger (instructions SQL)
  -- Accès aux valeurs : NEW.colonne / OLD.colonne
END;
Mot-cléSignificationDisponible
NEW.colonneNouvelle valeur (après modification)INSERT, UPDATE
OLD.colonneAncienne valeur (avant modification)UPDATE, DELETE
BEFORES'exécute avant l'opération (peut modifier NEW)INSERT, UPDATE, DELETE
AFTERS'exécute après l'opérationINSERT, UPDATE, DELETE

Exemple 1 — Journaliser les suppressions

-- Table de log
CREATE TABLE log_suppressions (
  id          INT AUTO_INCREMENT PRIMARY KEY,
  id_client   INT,
  nom         VARCHAR(100),
  supprime_le DATETIME DEFAULT NOW()
);

-- Trigger AFTER DELETE
CREATE TRIGGER trg_log_suppression_client
  AFTER DELETE ON clients
  FOR EACH ROW
BEGIN
  INSERT INTO log_suppressions (id_client, nom)
  VALUES (OLD.id_client, OLD.nom);
END;

Exemple 2 — Valider avant insertion (BEFORE INSERT)

DELIMITER $$

CREATE TRIGGER trg_check_age
  BEFORE INSERT ON clients
  FOR EACH ROW
BEGIN
  IF NEW.age < 0 OR NEW.age > 150 THEN
    SIGNAL SQLSTATE '45000'
      SET MESSAGE_TEXT = 'Âge invalide : doit être entre 0 et 150';
  END IF;
END$$

DELIMITER ;

Exemple 3 — Mettre à jour automatiquement (BEFORE UPDATE)

DELIMITER $$

CREATE TRIGGER trg_maj_date_modif
  BEFORE UPDATE ON clients
  FOR EACH ROW
BEGIN
  SET NEW.date_modification = NOW();
END$$

DELIMITER ;

Gestion des triggers

-- Voir tous les triggers de la base
SHOW TRIGGERS;
SHOW TRIGGERS FROM ma_base;

-- Supprimer un trigger
DROP TRIGGER IF EXISTS trg_log_suppression_client;
⚠️
Points importants :
• DELIMITER $$ est nécessaire pour les triggers multi-instructions (pour que MySQL ne confonde pas le ; du trigger avec la fin de la commande)
• Un trigger BEFORE peut modifier NEW.colonne pour changer la valeur avant insertion
• Un trigger AFTER ne peut pas modifier les données de la ligne concernée
• SIGNAL SQLSTATE '45000' permet de lever une erreur personnalisée
SQL 05 LCD — GRANT et REVOKE (Gestion des droits)

Le LCD (Langage de Contrôle des Données / DCL) gère les droits d'accès des utilisateurs à la base de données.

Créer et gérer des utilisateurs

-- Créer un utilisateur
CREATE USER 'alice'@'localhost' IDENTIFIED BY 'MotDePasse!123';
CREATE USER 'bob'@'%'          IDENTIFIED BY 'Secret456!';  -- % = tout hôte

-- Modifier le mot de passe
ALTER USER 'alice'@'localhost' IDENTIFIED BY 'NouveauMDP!';

-- Supprimer un utilisateur
DROP USER 'bob'@'%';

-- Voir tous les utilisateurs
SELECT user, host FROM mysql.user;
💡
'user'@'localhost' : connexion uniquement depuis la machine serveur
'user'@'%' : connexion depuis n'importe quel hôte
'user'@'192.168.1.%' : connexion depuis un sous-réseau spécifique

GRANT — Accorder des droits

-- Accorder des droits spécifiques sur une table
GRANT SELECT, INSERT
  ON ma_base.clients
  TO 'alice'@'localhost';

-- Accorder des droits sur toutes les tables d'une base
GRANT SELECT, INSERT, UPDATE, DELETE
  ON ma_base.*
  TO 'bob'@'%';

-- Accorder TOUS les droits sur une base
GRANT ALL PRIVILEGES ON ma_base.*
  TO 'admin'@'localhost';

-- Accorder tous les droits sur toutes les bases (super admin)
GRANT ALL PRIVILEGES ON *.*
  TO 'superadmin'@'localhost'
  WITH GRANT OPTION; -- peut lui-même accorder des droits

-- Appliquer les changements
FLUSH PRIVILEGES;
PrivilègeDescription
SELECTLire les données
INSERTInsérer des données
UPDATEModifier des données
DELETESupprimer des données
CREATECréer des tables/bases
DROPSupprimer des tables/bases
ALTERModifier la structure
INDEXGérer les index
TRIGGERGérer les triggers
CREATE VIEWCréer des vues
EXECUTEExécuter des procédures stockées
ALL PRIVILEGESTous les droits

REVOKE — Retirer des droits

-- Retirer des droits spécifiques
REVOKE INSERT, UPDATE
  ON ma_base.clients
  FROM 'alice'@'localhost';

-- Retirer tous les droits sur une base
REVOKE ALL PRIVILEGES ON ma_base.*
  FROM 'bob'@'%';

-- Voir les droits d'un utilisateur
SHOW GRANTS FOR 'alice'@'localhost';
⚠️
Principe du moindre privilège : accorder uniquement les droits nécessaires et rien de plus. Un compte d'application web n'a généralement besoin que de SELECT, INSERT, UPDATE, DELETE, jamais de DROP ou ALL PRIVILEGES.
SQL 06 LCD — Vues (Views)

Une vue est une requête SELECT sauvegardée sous forme d'objet de base de données. Elle se comporte comme une table virtuelle. Les données ne sont pas dupliquées, la vue exécute la requête à chaque fois qu'on l'interroge.

✅ Avantages

Simplifie les requêtes complexes, cache la complexité, sécurise l'accès (montrer seulement certaines colonnes), réutilisabilité.

⚠️ Limites

Pas d'index propre, pas de paramètre, parfois non modifiables, peut être moins performante qu'une table physique.

CREATE VIEW — Créer une vue

-- Vue simple : clients actifs avec leurs commandes
CREATE VIEW v_clients_actifs AS
  SELECT
    c.id_client,
    c.nom,
    c.prenom,
    c.email,
    c.ville,
    COUNT(cmd.id_commande) AS nb_commandes,
    SUM(cmd.montant)       AS total_achats
  FROM clients c
  LEFT JOIN commandes cmd ON c.id_client = cmd.id_client
  WHERE c.actif = TRUE
  GROUP BY c.id_client;

-- Utiliser la vue exactement comme une table
SELECT * FROM v_clients_actifs
WHERE ville = 'Paris'
ORDER BY total_achats DESC;

Vue pour la sécurité — Masquer des colonnes sensibles

-- Vue sans colonnes sensibles (mot de passe, données bancaires…)
CREATE VIEW v_clients_publics AS
  SELECT id_client, nom, prenom, ville
  FROM clients;

-- Donner accès à la vue seulement (pas à la table directe)
GRANT SELECT ON ma_base.v_clients_publics TO 'stagiaire'@'localhost';

Modifier et supprimer une vue

-- Modifier une vue existante
CREATE OR REPLACE VIEW v_clients_actifs AS
  SELECT id_client, nom, prenom, email
  FROM clients
  WHERE actif = TRUE;

-- Supprimer une vue
DROP VIEW IF EXISTS v_clients_actifs;

-- Voir toutes les vues
SHOW FULL TABLES WHERE Table_type = 'VIEW';

Vues modifiables et WITH CHECK OPTION

-- Vue modifiable (simple, sans agrégation ni jointure)
CREATE VIEW v_clients_lyon AS
  SELECT * FROM clients
  WHERE ville = 'Lyon'
WITH CHECK OPTION; -- empêche d'insérer un client d'une autre ville via cette vue

-- On peut insérer/modifier via la vue
INSERT INTO v_clients_lyon (nom, prenom, email, ville)
VALUES ('Test', 'User', 'test@mail.fr', 'Lyon'); -- ✅ OK

INSERT INTO v_clients_lyon (nom, prenom, email, ville)
VALUES ('Test', 'User', 'test@mail.fr', 'Paris'); -- ❌ ERREUR (WITH CHECK OPTION)

CYBER 01 RGPD & Identité numérique de l'organisation

Le RGPD (Règlement Général sur la Protection des Données) est un règlement européen entré en vigueur le 25 mai 2018. Il encadre la collecte, le traitement et le stockage des données personnelles.

Les 6 principes fondamentaux du RGPD

1️⃣ Licéité, loyauté, transparence

Les données doivent être traitées de façon légale, loyale et transparente pour la personne concernée.

2️⃣ Limitation des finalités

Les données doivent être collectées pour des finalités déterminées, explicites et légitimes. Pas de réutilisation à d'autres fins.

3️⃣ Minimisation des données

Collecter uniquement les données strictement nécessaires à la finalité. Pas de collecte excessive.

4️⃣ Exactitude

Les données doivent être exactes et mises à jour. Les données inexactes doivent être effacées ou rectifiées.

5️⃣ Limitation de la conservation

Les données ne peuvent être conservées plus longtemps que nécessaire à la finalité du traitement.

6️⃣ Intégrité et confidentialité

Sécurité appropriée : protection contre le traitement non autorisé, la perte ou la destruction accidentelle.

Le DPO — Délégué à la Protection des Données

Le DPO (Data Protection Officer) est le garant de la conformité RGPD. Il est obligatoire pour :

  • Les autorités et organismes publics
  • Les organismes dont l'activité principale implique un suivi régulier et systématique des personnes à grande échelle
  • Les organismes traitant des données sensibles (santé, origine ethnique, religion…) à grande échelle
💡
Le DPO doit être indépendant et ne peut pas être sanctionné pour l'exercice de ses fonctions. Il est l'interlocuteur de la CNIL.

Notification des violations de données

En cas de violation de données (fuite, cyberattaque…) :

  • Notifier la CNIL dans les 72h suivant la découverte
  • Informer les personnes concernées si la violation présente un risque élevé pour leurs droits

Sanctions RGPD

Niveau 1 — Infractions mineures

Jusqu'à 10 millions d'euros ou 2% du CA mondial annuel (le plus élevé des deux).

Niveau 2 — Infractions graves

Jusqu'à 20 millions d'euros ou 4% du CA mondial annuel (le plus élevé des deux).

Le chiffrement — Principe fondamental

🔑 Chiffrement symétrique

La même clé est utilisée pour chiffrer et déchiffrer. Rapide mais nécessite d'échanger la clé de façon sécurisée.

AlgorithmeTaille de cléUsage
AES-128128 bitsStandard actuel, suffisant
AES-256256 bitsTrès sécurisé, recommandé
DES56 bits❌ Obsolète (cassé en 1999)
3DES168 bits⚠️ Déprécié

🔐 Chiffrement asymétrique (clé publique / clé privée)

Deux clés mathématiquement liées : clé publique (partageable) pour chiffrer, clé privée (secrète) pour déchiffrer. Plus lent mais résout le problème d'échange de clé.

AlgorithmeUsage
RSA (2048 ou 4096 bits)Chiffrement, signature, échange de clés SSL/TLS
ECDSA / EdDSASignatures numériques (plus compact que RSA)
Diffie-HellmanÉchange de clés sécurisé

HTTPS / TLS — Sécuriser les communications web

TLS (Transport Layer Security) chiffre les communications entre le navigateur et le serveur. Fonctionnement simplifié :

  1. Le client demande une connexion sécurisée
  2. Le serveur envoie son certificat SSL/TLS (contient sa clé publique)
  3. Le client vérifie le certificat auprès d'une autorité de certification (CA)
  4. Les deux parties échangent une clé de session symétrique (via Diffie-Hellman)
  5. La communication est chiffrée avec AES
✅
TLS 1.3 est la version actuelle recommandée. Les versions TLS 1.0, 1.1 et SSL 2.0/3.0 sont obsolètes et vulnérables (POODLE, BEAST, etc.).

VPN — Réseau Privé Virtuel

Un VPN crée un tunnel chiffré entre le client et le serveur, masquant le trafic des écoutes intermédiaires.

Protocole VPNSécuritéUsage
OpenVPN⭐⭐⭐⭐⭐Standard entreprise, open source
WireGuard⭐⭐⭐⭐⭐Moderne, très rapide, code minimal
IPSec/IKEv2⭐⭐⭐⭐VPN mobile, natif sur iOS/Android
PPTP⭐❌ Obsolète, cassé

Authentification Multifacteur (MFA/2FA)

Le MFA combine au moins 2 facteurs d'authentification parmi :

  • 🧠 Ce que tu sais : mot de passe, PIN, question secrète
  • 📱 Ce que tu possèdes : smartphone (OTP), clé physique (YubiKey), carte à puce
  • 👁️ Ce que tu es : empreinte digitale, reconnaissance faciale, iris
✅
Le MFA réduit de 99,9% le risque de compromission d'un compte, même si le mot de passe est volé (source Microsoft).
CYBER 03 Intégrité des données

L'intégrité garantit que les données n'ont pas été modifiées de façon non autorisée. C'est le I de la triade CIA.

Les fonctions de hachage (Hash)

Une fonction de hachage transforme des données en une empreinte numérique de taille fixe. Elle est à sens unique (irréversible) et déterministe.

AlgorithmeTaille du hashSécurité
MD5128 bits (32 hex)❌ Cassé, collisions connues — à éviter
SHA-1160 bits (40 hex)❌ Obsolète — déprécié depuis 2017
SHA-256256 bits (64 hex)✅ Standard actuel
SHA-512512 bits (128 hex)✅ Très sécurisé
SHA-3Variable✅ Algorithme de remplacement (nouvelle famille)
bcrypt / Argon2Variable✅ Recommandé pour les mots de passe
💡
Propriétés d'une bonne fonction de hachage :
• Déterministe : même entrée → même hash
• Rapide à calculer
• Résistance aux préimages : impossible de retrouver les données depuis le hash
• Résistance aux collisions : deux entrées différentes ne doivent pas donner le même hash
• Effet avalanche : un seul bit changé modifie tout le hash

Salage des mots de passe (Salt)

Le sel est une valeur aléatoire ajoutée au mot de passe avant le hachage, pour contrer les attaques par rainbow table.

-- Sans sel : même MDP → même hash (vulnérable aux rainbow tables)
SHA256("password123") → e2b05... ← toujours identique

-- Avec sel aléatoire : hashes différents même pour le même MDP
SHA256("password123" + "aZ9kT!") → f8c2a...
SHA256("password123" + "rP3mX#") → 9d1b7...  ← différent !

Signature numérique

La signature numérique garantit :

  • Authentification : l'expéditeur est bien celui qu'il prétend être
  • Intégrité : le document n'a pas été modifié en transit
  • Non-répudiation : l'expéditeur ne peut pas nier avoir signé
💡
Fonctionnement :
1. Alice calcule le hash du document
2. Elle chiffre ce hash avec sa clé privée → c'est la signature
3. Bob déchiffre la signature avec la clé publique d'Alice
4. Il recalcule le hash du document reçu
5. Si les deux hash correspondent → intégrité et authenticité confirmées

HMAC — Authentification de message

Le HMAC (Hash-based Message Authentication Code) est un hash calculé avec une clé secrète partagée. Il garantit à la fois l'intégrité et l'authenticité sans utiliser de clé privée/publique.

HMAC-SHA256(clé_secrète, message) → code_d_authentification

Utilisé dans les JWT (JSON Web Tokens), les API authentifiées, etc.

PKI — Infrastructure à Clés Publiques

La PKI (Public Key Infrastructure) est l'ensemble des mécanismes permettant de gérer les certificats numériques.

ComposantRôle
Autorité de Certification (CA)Émet et signe les certificats (ex: Let's Encrypt, DigiCert)
Autorité d'Enregistrement (RA)Vérifie l'identité avant d'émettre un certificat
Certificat X.509Document numérique liant une clé publique à une identité
CRL / OCSPListes de certificats révoqués (compromis)

La disponibilité garantit que les systèmes et données sont accessibles quand les utilisateurs en ont besoin. C'est le D de la triade CIA.

La règle 3-2-1 pour les sauvegardes

3️⃣ — 3 copies

Conserver au moins 3 copies des données (1 originale + 2 sauvegardes).

2️⃣ — 2 supports

Sur 2 types de supports différents (ex: disque local + bande magnétique).

1️⃣ — 1 hors site

Au moins 1 copie hors site (cloud, datacenter distant, coffre-fort).

⚠️
Une sauvegarde non testée n'est pas une sauvegarde ! Tester régulièrement la restauration est indispensable. Un ransomware peut aussi chiffrer les sauvegardes réseau — garder une copie offline et air-gapped.

Haute disponibilité — Solutions techniques

⚖️ Load Balancer

Répartit le trafic entre plusieurs serveurs pour éviter la surcharge et assurer la continuité en cas de panne d'un serveur.

🔄 Clustering

Groupe de serveurs fonctionnant ensemble. En cas de panne, un autre nœud prend le relais automatiquement (failover).

🪞 Réplication

Copie en temps réel des données sur un serveur secondaire (réplication master-slave en BDD).

☁️ Cloud / Scalabilité

Augmentation automatique des ressources en cas de pic de charge (autoscaling AWS, Azure, GCP).

Monitoring & Supervision

Pour garantir la disponibilité, il faut superviser en permanence :

  • Infrastructure : CPU, RAM, disque, réseau (Nagios, Zabbix, Prometheus)
  • Applications : temps de réponse, erreurs, disponibilité (Grafana, Datadog)
  • Logs : centralisation et analyse (ELK Stack, SIEM)
  • Alertes : notification immédiate en cas d'anomalie (email, SMS, Slack)

Gestion des correctifs (Patch Management)

Les vulnérabilités des logiciels sont la principale porte d'entrée des attaquants. La gestion des correctifs consiste à :

  1. Inventorier tous les logiciels et leurs versions
  2. Surveiller les CVE (Common Vulnerabilities and Exposures)
  3. Tester les correctifs en environnement de pré-production
  4. Déployer rapidement les correctifs critiques
  5. Vérifier l'application des mises à jour
🚨
EternalBlue (WannaCry, 2017) et Log4Shell (2021) ont exploité des failles sur des systèmes non mis à jour. La mise à jour est la mesure de sécurité la plus importante et la plus négligée.

Injection SQL — En détail

-- VULNÉRABLE : concaténation directe de l'entrée utilisateur
// PHP vulnérable :
$query = "SELECT * FROM users WHERE login = '" . $_GET['user'] . "' AND password = '" . $_GET['pass'] . "'";

-- Attaque : user = admin'-- (le -- commente le reste de la requête)
-- Requête exécutée : SELECT * FROM users WHERE login = 'admin'--' AND ...
-- Résultat : connexion sans mot de passe !

-- SÉCURISÉ : requête préparée (PDO)
// PHP sécurisé :
$stmt = $pdo->prepare("SELECT * FROM users WHERE login = ? AND password = ?");
$stmt->execute([$_GET['user'], $_GET['pass']]);
✅
Remédiations contre l'injection SQL :
1. Requêtes préparées (PDO, PreparedStatement) — solution principale
2. ORM (Doctrine, Eloquent, Hibernate) — abstraction sécurisée
3. Validation des entrées — whitelist des valeurs attendues
4. Principe du moindre privilège — compte BDD sans DROP/ALTER
5. WAF — dernière ligne de défense

XSS — Cross-Site Scripting

L'attaquant injecte du code JavaScript malveillant dans des pages vues par d'autres utilisateurs.

Type XSSFonctionnement
Reflected XSSLe script est dans la requête HTTP et reflété immédiatement dans la réponse
Stored XSSLe script est stocké en BDD et exécuté pour tous les visiteurs
DOM-based XSSManipulation du DOM côté client sans passer par le serveur
✅
Remédiations XSS :
• Échappement des sorties : htmlspecialchars() en PHP, encode en Angular/React
• CSP (Content Security Policy) : entête HTTP limitant les sources de scripts
• HttpOnly sur les cookies : empêche JS d'accéder aux cookies de session
• Bibliothèques de sanitisation (DOMPurify, OWASP Java HTML Sanitizer)

CSRF — Cross-Site Request Forgery

Un site malveillant force le navigateur d'un utilisateur authentifié à effectuer une action sur un site légitime à son insu.

✅
Remédiations CSRF :
• Token CSRF : jeton aléatoire dans chaque formulaire, vérifié côté serveur
• SameSite cookie : attribut Strict ou Lax sur les cookies de session
• Vérification de l'origine : contrôle des entêtes Origin/Referer
